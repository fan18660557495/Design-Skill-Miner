# 怎么搭一个类似的设计沉淀 Agent

这篇文档不是教你逐行看代码，而是想用比较容易理解的方式，讲清楚：

**如果你也想做一个类似的 Agent，应该从哪里开始想。**

先说明一下当前仓库的状态：

- 已经包含 skills 层（定义、路由、执行）
- 已经包含策略层（根据审校原因自动决定下一步）
- 已经包含项目记忆层（跨运行复用经验）
- 已经包含发布审批闸门（发布前人工确认）

下面这套方法不是“未来规划”，而是这套仓库正在使用的搭建方式。

## 先别急着想“模型”

很多人一开始做 Agent，第一反应是：

- 用什么模型
- 用不用 RAG
- 要不要多智能体

这些问题当然重要，但不是第一步。

第一步更应该问的是：

- 我要帮谁解决问题
- 我要沉淀什么内容
- 什么样的内容才算有长期价值
- 哪一步应该交给机器
- 哪一步必须交给人

如果这几个问题没想清楚，再强的模型也只会把混乱变得更像“看起来有道理的混乱”。

## 这个项目的方法很简单

你可以把它总结成一句话：

**先找到值得沉淀的内容，再把它整理清楚，而不是直接让模型替你下结论。**

## 第一步：定义沉淀对象

先明确你到底要沉淀什么。

在这个项目里，沉淀对象是：

- 页面结构经验
- 组件规范
- 交互反馈
- 文案规则
- 样式系统口径

如果你做的是别的 Agent，沉淀对象也许会变成：

- 销售沟通话术
- 客服处理流程
- 研究结论模板
- 运营投放经验

先把对象说清楚，后面所有步骤才会稳定。

## 第二步：定义“什么才值得留下”

不是所有讨论都值得进入沉淀。

这个项目依赖的是几个非常朴素的判断：

- 是否反复出现
- 是否有足够证据
- 是否已经相对稳定
- 是否能整理成清楚规则

这一步非常关键。

因为 Agent 最容易犯的错，不是“看不见重要内容”，而是“把不该沉淀的东西留下来”。

## 第三步：把流程拆开

不要让一个步骤同时承担太多事情。

在这个项目里，流程被拆成：

1. 读材料
2. 找项目
3. 过滤内容
4. 找重复话题
5. 整理规则
6. 做审校
7. 可选润色
8. 生成草稿
9. 人工确认后发布

拆开之后，每一步都更容易解释，也更容易调整。

## 第四步：让模型只做它真正擅长的事

模型很适合做这些事：

- 把一句话改写得更清楚
- 把几条规则写得更一致
- 把表达整理得更容易读

模型不太适合直接替你决定：

- 哪条规则才是真的团队共识
- 哪个方向已经定了
- 哪个临时讨论应该升格成正式规范

所以更稳妥的做法通常是：

**先找候选，再润色表达。**

## 第五步：一定要保留人工确认

这是最容易被忽视的一点。

如果你的 Agent 面向的是“规范”“经验”“知识沉淀”这类内容，就几乎一定要保留人工确认。

因为这类内容不是只要“写得像答案”就行，而是要真的能代表团队默认做法。

所以这个项目输出的是：

- 洞察
- 审校
- 草稿

而不是“最终规范”。

## 第六步：前台只保留真正需要决定的东西

如果你的使用者是设计师，就不要把前台做成参数面板。

设计师更关心的是：

- 我要处理哪个项目
- 要不要启用润色
- 结果现在跑到哪一步了
- 草稿能不能继续改

而不是：

- 模型名字是什么
- 超时时间是多少
- 审校阈值是不是 0.6

这些内容更适合放到后台配置。

## 第七步：把“发布”放在最后

如果你的 Agent 会影响团队知识库、技能库、规范库，发布一定不要和生成绑死。

正确的节奏通常是：

1. 先生成
2. 再看草稿
3. 再确认
4. 最后发布

这样哪怕前面某一步判断不够好，也不会直接污染正式内容。

## 一个简单的通用模板

如果你想照着做一个类似 Agent，可以先按这个模板思考：

### 输入层

- 你要读什么材料
- 材料从哪里来

### 筛选层

- 哪些内容是目标内容
- 哪些内容应该丢掉

### 发现层

- 怎么判断哪些话题值得留下
- 什么算“重复”

### 整理层

- 最后想输出成什么形式
- 是规则、模板、摘要，还是清单

### 质检层

- 什么情况下不应该进入最终结果

### 人工层

- 人在哪一步接管
- 人可以修改什么

### 发布层

- 是直接发布，还是先进暂存区

## 这个项目最值得复用的，不是代码细节

最值得复用的是这几个原则：

- 先定义目标，再选工具
- 先找信号，再做润色
- 先出草稿，再做发布
- 前台尽量轻，后台承接复杂度
- 把人工确认当成流程的一部分，而不是补救手段

如果你按这个思路去做，即使最后不是做“设计沉淀 Agent”，而是做别的知识整理 Agent，也会更稳。
