# 这个 Agent 的完整形态

这份文档用非工程化的表达，说明 Design Skill Miner 现在到底是什么。

一句话定义：

**它是一个 Workflow-first 的设计沉淀 Agent，带 skills、策略循环、项目记忆和发布审批闸门。**

它不是“一次性总结器”，也不是“自动定稿器”。  
它是一个会持续整理、评估、重试，并把最终决定留给人的系统。

## 它在做的事

你可以把它想成一个设计团队的“沉淀执行员”：

- 把历史讨论收集起来
- 找出反复出现的主题
- 按合适的 skill 来组织内容
- 对质量做审校和风险标注
- 根据风险自动决定下一步（继续收集、换策略、或结束）
- 输出可编辑草稿
- 发布前要求人工确认

## 它为什么不只叫 workflow

它确实有稳定流程，但现在还多了 4 个 Agent 核心能力：

1. **目标层**
每次运行都有明确目标、成功标准和最大循环次数。

2. **策略层**
它会根据审校原因码做决策，而不是固定一条线跑到底。

3. **记忆层**
它会记住项目历史运行表现，并在下一次运行前先读取这些记忆。

4. **闸门层**
自动生成和正式发布被分开，发布默认需要显式批准。

## 当前运行顺序

当前的完整链路是：

1. 初始化目标（本次要沉淀什么）
2. 读取项目记忆（历史风险、偏好 skill、建议阈值）
3. 扫描会话并提炼候选
4. 路由到合适的 skill（或使用手动指定）
5. 进行审校并标注主要风险原因
6. 触发策略决策（补扫、换 skill、或结束）
7. 在循环次数内重复执行 3~6
8. 应用 skill 输出策略生成草稿结构
9. 可选执行 LLM 润色
10. 产出报告和草稿
11. 写回项目记忆
12. 由人确认后再发布

你可以把第 3~7 步理解为 Agent 的“决策循环”。

## 为什么要有 skills

skills 在这里不是“额外功能”，而是主流程的一部分。

它负责定义三件事：

- 当前任务该重点关注哪些分类
- 审校阈值应该怎么设
- 草稿应该优先呈现什么内容

这样同样的会话材料，在不同目标下可以走不同沉淀策略，而不是一套模板覆盖所有场景。

## 为什么要有记忆

没有记忆，Agent 每次都会把项目当成新项目。

有记忆后，它可以利用历史运行经验：

- 经常被拒绝的标题会降低优先级
- 在这个项目里效果更稳定的 skill 会优先考虑
- 证据不足经常出现时，会自动提高下一轮收集阈值

这让它从“单次执行器”变成“持续学习型执行器”。

## 为什么发布要审批

规范沉淀最怕的是把阶段性讨论直接写进正式知识库。

所以系统里明确分成两步：

1. 先生成草稿（可编辑）
2. 人工批准后发布

这不是增加流程负担，而是保护沉淀质量。

## 代码映射（按职责，不按文件名堆砌）

- 目标与循环编排：`agent.py`
- 风险判断与原因码：`review.py`
- skills 定义与路由：`skills/*.json`、`skill_registry.py`、`skill_router.py`、`skill_executor.py`
- 策略决策：`tool_policy.py`
- 项目记忆：`memory.py`
- 可选语言润色：`llm.py`
- 报告与草稿：`report.py`、`draft_skill.py`
- 发布与审批接口：`web_support.py`、`publish_skill.py`

## 给开源读者的重点

如果你想复刻类似 Agent，最值得学的不是某个模型参数，而是这套分层：

- 稳定流程层：保证可解释、可复现
- 策略层：让系统能自动调整
- 记忆层：让系统能跨运行学习
- 审批层：保证最终质量边界

这 4 层同时存在，才能从“好用 workflow”走向“可持续的团队 Agent”。
