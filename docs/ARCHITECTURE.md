# 这个 Agent 是怎么工作的

这篇文档不是写给只看代码的人，而是写给想真正理解这个 Agent 为什么这样设计的人。

如果你是设计师，可以把它当成“这个工具背后的工作方式说明”。

如果你是想复刻一个类似 Agent 的人，可以把它当成“这个项目为什么这样拆”的解释。

## 先说结论

这个 Agent 最重要的不是“大模型有多强”，而是它把整件事拆成了几个很清楚的小步骤。

它没有试图让模型直接从一堆聊天里一次性写出最终规范，而是把工作拆成：

1. 先收集材料
2. 再找重复内容
3. 再整理成候选规则
4. 再做质量判断
5. 最后才生成草稿

这样做的好处是：

- 结果更稳定
- 更容易解释为什么会得到这份草稿
- 更容易做人工复核
- 更适合长期持续沉淀

## 用设计师能理解的话说，它在做什么

你可以把这个 Agent 想成一个设计研究助理。

它平时不会自己发明规范，而是做这些事：

- 把旧讨论翻出来
- 找到那些你们已经反复说过的内容
- 帮你把零散话语整理成更像规则的表达
- 提醒你哪些内容其实还不够稳定
- 最后交给你一份可以继续改的草稿

所以它的角色不是“替你设计”，而是“替你整理已经发生过的设计判断”。

## 为什么不能一上来就让模型直接写

这是很多人做 Agent 时最容易踩的坑。

如果一上来就把所有聊天记录丢给模型，让它“总结成规范”，通常会出现这些问题：

- 很容易把一次性讨论误当成长期规则
- 很容易把还没定的方向写成结论
- 生成的内容看起来很完整，但和真实项目语境不完全一致
- 很难解释为什么它会写成这样

这个项目故意避开了这种方式。

它的思路是：

**先让流程帮你找到“有重复、有证据”的内容，再让模型去润色表达。**

## 它的大致工作顺序

## 1. 收集材料

第一步是读取本地会话导出。

这些会话是原材料，但原材料本身并不等于规则。

里面会同时包含：

- 真正有价值的设计讨论
- 临时问题
- 偶发 bug
- 还没定下来的方向
- 与设计沉淀无关的内容

所以这一步只是“把材料拿到手”，还不是开始沉淀。

## 2. 找到和某个项目有关的内容

真实使用时，一个人可能同时聊很多项目。

如果不先按项目分开，最后得到的规则会非常混乱。

所以这个 Agent 会先尽量判断：

- 这段讨论属于哪个项目
- 当前用户想看的是哪个项目

这样做的目的，是让输出尽量保持在同一个上下文里。

## 3. 挑出设计相关讨论

接下来，它会尽量把真正可能有沉淀价值的内容挑出来。

重点关注的是：

- 页面结构
- 组件写法
- 交互反馈
- 样式规则
- 文案表达

这一步其实很关键，因为它决定了后面是不是会被很多噪音带偏。

## 4. 找“重复出现”的主题

为什么要找重复？

因为真正适合沉淀的内容，通常不会只出现一次。

如果同一个问题在多个会话里反复被提到，它更可能代表：

- 团队已经形成了某种习惯
- 这个问题本身非常重要
- 这个地方缺少明确默认做法

所以“重复”在这里不是缺点，反而是最有价值的信号。

## 5. 把重复主题整理成候选规则

真实聊天里的表达通常很碎。

比如大家会说：

- “这个地方别再用这种写法了”
- “这里上次不是统一过吗”
- “这个弹窗还是走原来的结构”

这些话对当时的讨论很有帮助，但不适合作为长期沉淀。

于是这个 Agent 会尝试把它们整理成：

- 更完整的标题
- 更清楚的摘要
- 更像规则的表达

这样后面的人看到的，就不只是聊天碎片，而是可继续使用的候选内容。

## 6. 做一轮质量检查

这一步是这个 Agent 非常重要的一层。

它不会默认所有候选都值得留下，而是会看一些很实际的问题：

- 证据是不是太少
- 出现次数是不是太低
- 规则是不是太空
- 不同候选是不是其实在说同一件事

换句话说，这一步在回答：

**“这条内容现在真的适合沉淀吗？”**

## 7. 可选润色

如果你开启了润色，它会再把候选规则写得更清楚一些。

这里要注意：

润色的重点是“把话写清楚”，不是“重新发明一套新规则”。

所以更准确地说，它在做的是：

- 统一表达方式
- 让摘要更易读
- 让规则更像规范条目

而不是替你做最终业务判断。

## 8. 生成草稿

最后，它不会直接生成正式规范，而是生成草稿。

这是整个项目非常明确的设计取向：

**先草稿，再确认。**

因为设计沉淀本身就不适合完全自动化。

真正可靠的流程应该始终保留一层人工判断。

## 为什么 Web 端现在做得很轻

这个项目是给设计师看的，所以前台故意尽量轻。

前台只保留用户真正需要决定的事情，比如：

- 选哪个项目
- 要不要启用润色
- 要不要继续发布

而像这些内容，尽量放到后台配置里：

- 模型
- API 地址
- 密钥环境变量
- 审校阈值

这样做的原因很简单：

设计师需要的是“好用”，不是“看见所有参数”。

## 为什么要先生成草稿再发布

如果一个 Agent 一运行就把内容直接写进正式规范，很容易出现两个问题：

1. 阶段性结论被误当成长期规则
2. 错误规则进入正式知识库后，很难被发现

所以这个项目把“发布”放在后面，而且默认先进入暂存区。

这一步虽然看起来多了一道手续，但实际是在保护沉淀质量。

## 如果你想照着做一个类似的 Agent

最值得借鉴的，不是具体代码，而是这个拆法。

你完全可以把自己的 Agent 也按这个思路来搭：

### 1. 先想清楚你要沉淀什么

不是所有聊天都值得沉淀。

你要先定义：

- 你到底要提炼的是页面规则、服务流程、销售话术，还是别的东西
- 什么样的话题才算“值得留下”

### 2. 不要直接让模型做全部工作

更稳妥的方式通常是：

- 先用固定流程找候选
- 再用模型做语言整理
- 最后让人做判断

### 3. 一定要有“质量检查”这一层

如果没有这一层，Agent 很容易把偶发问题、一次性讨论、临时决定都写进结果里。

### 4. 输出一定要能继续编辑

如果输出只是一个一次性回答，长期价值会很低。

更好的做法是：

- 输出成草稿
- 允许人继续改
- 允许再次发布

### 5. 尽量把前台做轻

如果你的使用者不是工程师，而是设计师、运营、研究员，那前台最好只保留真正需要决定的内容。

参数越多，负担越重。

## 对外开源时，别人最该学到什么

如果别人只是把这个仓库跑起来，那只是“用了一个工具”。

但如果别人能从这个仓库学会下面这些事，开源才更有价值：

- 不要一上来就全交给模型
- 要先定义什么内容值得沉淀
- 要把“重复”当成重要信号
- 要把“质量检查”单独做成一步
- 要把“草稿”和“正式发布”分开
- 要让前台尽量贴近真正的使用者

这其实就是这个项目最想传达的方法论。
