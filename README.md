# Design Skill Miner

`design-skill-miner` 是一个面向设计师的本地 Agent。

它的作用不是单纯“总结聊天记录”，而是把你和 AI、团队成员在日常讨论里反复说到的设计判断，慢慢整理成一套可以复用、可以继续编辑、可以逐步沉淀的 Skill 草稿。

你可以把它理解成一个“设计沉淀助手”：

- 平时它帮你从大量对话里找重复出现的设计规则
- 运行后它会把这些规则整理成更容易阅读的草稿
- 最后你再决定哪些内容值得保留，哪些内容只是阶段性讨论

它适合的不是“一次性产出”，而是“长期沉淀”。

## 为什么会有这个 Agent

很多设计团队都会遇到同一个问题：

- 同一个按钮规范反复说
- 同一种弹窗写法反复改
- 同一套状态提示反复统一
- 同一类页面结构反复讨论
- 同一类文案规则反复被纠正

这些内容本来都是经验，但如果不整理，它们只会散落在聊天里。

时间久了，团队会出现几个很典型的现象：

- 新人不知道默认做法是什么
- 同样的问题一次次重复讨论
- 每个人脑子里都有一套“应该怎么做”，但没有统一版本
- 过去明明已经讨论过的内容，后面又从头来过

这个 Agent 想解决的就是这件事：

**把零散的设计讨论，慢慢变成团队真正能复用的规则草稿。**

## 它到底是什么

从结果上看，它会产出几类东西：

- 规则洞察：哪些话题反复出现了
- 审校结果：哪些内容更稳定，哪些内容还不适合沉淀
- Skill 草稿：把讨论整理成可编辑文档

从工作方式上看，它是一条完整流程，而不是一个单点工具：

1. 读本地会话
2. 找到属于某个项目的讨论
3. 挑出设计相关内容
4. 找出那些反复出现的话题
5. 先选择或自动路由到最合适的沉淀 skill
6. 把话题整理成候选规则
7. 做一次质量检查，并标出主要风险原因
8. 根据风险自动决定下一步（补扫、换 skill、或结束）
9. 把运行结果写入项目记忆，给下一次运行做参考
10. 可选再用模型润色表达
11. 生成草稿
12. 发布前由人确认（默认人工闸门）

所以它不是：

- 普通聊天摘要器
- 自动写规范的黑盒
- 一键生成最终答案的工具

它更像一个：

**先帮你整理，再把决定权留给你的设计 Agent。**

## 它是怎么工作的

这里尽量不用太多技术词，按设计师能理解的方式讲。

### 第一步：先收集材料

它会去读本地会话导出，也就是你过去和 AI 或团队在某个项目里的对话记录。

这些记录本身很杂，里面会混有很多不是规范的内容，比如：

- 临时沟通
- 一次性需求
- 还没定的方向
- 偶发 bug 讨论

所以它不会把所有内容都直接当规则。

### 第二步：只挑设计相关内容

它会先把明显和设计沉淀无关的部分过滤掉，尽量把注意力放在这些内容上：

- 页面结构
- 组件写法
- 交互反馈
- 文案表达
- 样式系统

这一步的目的很简单：

**先把“素材池”缩小到真的值得沉淀的范围。**

### 第三步：找出重复出现的话题

真正有沉淀价值的内容，通常不是只出现一次。

比如：

- 同一个组件规范在不同天都被提到
- 同一类页面结构在多个页面里都反复讨论
- 同一种反馈文案在多个场景里被统一

这个 Agent 会去找这种“反复出现”的主题，因为只有重复，才更可能是团队真正需要的默认做法。

### 第四步：把讨论整理成规则候选

找到重复话题后，它不会直接把原话贴出来，而是会尝试整理成更像规则的表达。

比如不是只保留：

“这里感觉按钮应该更统一一点”

而是尽量整理成：

“同一层级按钮的主次关系要稳定，不要在同类页面里反复切换主按钮样式”

这一步非常重要，因为沉淀不是保存原话，而是把原话变成能复用的内容。

### 第五步：先选对 skill，再做审校

在进入审校前，Agent 会先决定这轮更适合哪一种沉淀 skill（或使用你手动指定的 skill）。

skill 不只是分类标签，它会一起带来：

- 这轮应该重点看哪些分类
- 审校阈值应该更严格还是更宽松
- 草稿里优先呈现哪些内容

然后才进入审校。

### 第六步：不是只打分，还要给出主要风险原因

不是所有候选都值得进入草稿。

这个 Agent 会做一轮质量检查，并明确风险原因码，比如：

- 这个话题是不是只出现过一次
- 它是不是证据太少
- 规则是不是太空
- 几条规则是不是其实在说同一件事

这一层的作用不是“替你做最终判断”，而是给后面的自动重规划提供依据。

### 第七步：进入自动重规划循环

如果这轮结果不理想，它不会立刻结束，而是根据策略层自动做下一步决策：

- 提高重复阈值再扫描一轮
- 切换到更匹配的 skill 再评估
- 达到目标分后结束循环

循环次数是可控的，默认不会无限跑。

### 第八步：写入项目记忆

每次运行结束后，Agent 会把关键结果写入项目记忆，例如：

- 哪类风险经常出现
- 哪个 skill 在这个项目里更稳定
- 哪些标题经常被拒绝

下一次运行会先读这些记忆，再决定初始策略。

### 第八点五步：保存草稿会写入“可审计学习记录”

现在除了运行结果记忆，Agent 还会记录你的草稿编辑偏好。

当你在前台点“保存草稿”时，系统会：

- 记录这次改动（写入本地 memory 数据库）
- 给改动打上可审计标签（例如更偏好示例、边界、可执行规则）
- 在下一次运行时把这些偏好作为记忆提示，影响草稿描述和策略

这意味着：

**你手动改过的草稿，后续运行会逐步“记住你的写法偏好”。**

说明：

- 这个学习是“有审计记录”的，不是黑盒
- 不会替你自动定稿，依然是“Agent 整理 + 人工确认”

### 第九步：可选润色

如果你开启了润色，它会再把规则表达写得更清楚一点。

这里要强调的是：

**润色的目标不是发明新规则，而是把已有规则写得更清楚、更像规范。**

所以它更像“整理语言”，不是“重新发明结论”。

### 第十步：生成草稿，而不是直接定稿

最后它会输出一份 Skill 草稿。

之所以是“草稿”，是因为设计沉淀本来就不应该完全自动化。

真正可靠的流程应该是：

**Agent 负责整理，人来负责确认。**

这也是这个项目最核心的理念之一。

## 它适合用在哪些场景

### 1. 团队已经讨论了很多，但没有正式规范

这是最适合的场景。

你们其实已经有很多共识，只是这些共识还散落在聊天里，没有整理出来。

这个 Agent 可以帮你把这些“隐性的共识”先提出来。

### 2. 团队经常和 AI 一起做设计讨论

如果你们已经大量用 AI 辅助页面、组件、文案、交互设计，就会产生很多高密度讨论记录。

这些记录本身就是最好的沉淀原料。

### 3. 你想慢慢形成项目自己的设计习惯

有些团队不想一开始就做很重的设计系统，而是希望先把“默认做法”沉淀出来。

这个 Agent 很适合做这件事，因为它输出的是草稿，而不是一套很重的制度。

### 4. 新人上手成本高

如果一个项目已经跑了很久，新人最常遇到的问题就是：

- 这个页面以前是怎么做的
- 这个组件为什么要这样写
- 这类提示语有没有统一口径

这些内容如果提前沉淀，新人会轻松很多。

## 它不适合什么场景

### 1. 你的对话记录本身很少

如果讨论量很少，它就找不到真正“重复”的东西。

### 2. 你的项目还在非常早期

如果很多东西都还没定，太早做沉淀反而会把临时想法误当成规则。

### 3. 你希望完全自动生成最终规范

这个项目不鼓励那样做。

它强调的是“先整理，再确认”，而不是“机器直接定稿”。

## 你可以怎么用它

## 方式一：用 Web 界面

这是最适合设计师的方式。

```bash
./run-local.sh serve --host 127.0.0.1 --port 8765
```

打开：

- [http://127.0.0.1:8765](http://127.0.0.1:8765)

你只需要做四件事：

1. 选择项目目录
2. 选择沉淀 skill（可选，不选就是自动）
3. 决定是否启用润色
4. 点击“启动智能体”

后面它会自己继续跑：

- 提炼规则
- 选择/路由 skill
- 做审校和自动重规划
- 写入项目记忆
- 生成草稿

你再去看结果、改草稿、决定要不要发布到暂存区。

## 方式二：用命令行

如果你更习惯命令行，可以直接这样跑：

```bash
design-skill-miner agent-mine /path/to/sessions \
  --cwd-prefix "/path/to/project" \
  --out ./agent-out \
  --skill-name my-design-skill
```

如果要启用润色：

```bash
design-skill-miner agent-mine /path/to/sessions \
  --cwd-prefix "/path/to/project" \
  --out ./agent-out \
  --skill-name my-design-skill \
  --enable-llm
```

## 它会输出什么

一次完整运行通常会生成：

```text
agent-out/
├── agent-run.json
├── draft/
│   ├── SKILL.md
│   ├── manifest.json
│   └── references/
└── reports/
    ├── insights.json
    ├── insights.md
    └── review.json
```

你可以这样理解这些文件：

- `insights.json`：更适合程序处理的规则候选
- `insights.md`：更适合人阅读的规则洞察
- `review.json`：这次结果值不值得继续用
- `draft/`：真正可以继续编辑的草稿
- `agent-run.json`：这次运行做了哪些步骤

## 如果你也想搭一个类似的 Agent

这个仓库不只是想把一个工具开源出来，更希望别人能看懂：

**一个设计沉淀 Agent，应该怎么搭。**

最重要的不是把所有能力都交给大模型，而是先把问题拆清楚。

你可以先把这样的 Agent 想成 6 个部分：

1. 输入材料
2. 候选筛选
3. 重复识别
4. 规则整理
5. 质量检查
6. 人工确认

也就是说，先别急着问“该用什么模型”，先问：

- 我到底要沉淀什么
- 什么样的讨论才算值得沉淀
- 我怎么判断它是重复而不是偶发
- 输出应该给谁看
- 哪一步必须人工确认

这个项目给出的核心做法是：

**稳定流程负责“找对内容”，skills + 决策循环负责“选对策略”，模型只负责“把话写清楚”。**

当前仓库的完整形态已经包含 skills 层和策略层：

- `skills/*.json` 定义每类沉淀任务的关注点、审校标准和草稿引导
- `skill_router.py` 按当前候选分布自动选择 skill（也支持手动指定）
- `skill_executor.py` 把 skill 的输出偏好应用到草稿生成
- `tool_policy.py` 根据审校原因码决定下一步动作（补扫、换 skill、结束）
- `memory.py` 记录历史运行结果，给下一次运行提供项目记忆

这里的“一小部分”，通常就是：

- 帮你把话写清楚
- 帮你把规则写得更像规范
- 帮你把表达变得更统一

而不是替你决定业务规则本身。

如果你想照着做，建议直接看：

- [docs/ARCHITECTURE.md](./docs/ARCHITECTURE.md)：讲这个 Agent 的整体结构
- [docs/HOW-TO-BUILD-THIS-AGENT.md](./docs/HOW-TO-BUILD-THIS-AGENT.md)：讲怎么搭一个类似的 Agent

## 项目结构

```text
design-skill-miner/
├── src/design_skill_miner/   # Agent 核心
├── web/                      # 本地 Web UI
├── docs/                     # 说明文档
├── README.md
├── CONTRIBUTING.md
├── SECURITY.md
├── CHANGELOG.md
└── .design-skill-miner.toml.example
```

如果只从理解角度看，这几个文件最重要：

- `agent.py`：负责把整条流程串起来
- `review.py`：负责先做一轮质量判断
- `llm.py`：负责可选的润色能力
- `skill_registry.py` / `skill_router.py` / `skill_executor.py`：负责 skills 的定义、选择和执行
- `tool_policy.py`：负责根据审校结果决定下一步动作
- `memory.py`：负责项目长期记忆（历史运行结果、偏好和风险）
- `draft_skill.py`：负责把结果写成草稿
- `web.py` / `web_support.py`：负责让设计师用得起来

## 按流程看，这些文件分别在做什么

如果你想快速讲清楚这个仓库，可以不要一上来就按文件名背，而是按“流程”来记。

### 1. 启动入口

这一层回答的是：

**用户怎么把这个 Agent 跑起来。**

- CLI 入口：`cli.py`
- 模块入口：`__main__.py`
- Web 服务入口：`web.py`
- 启动脚本：`run-local.sh`、`Start Design Skill Miner.command`、`start-design-skill-miner.sh`

你可以把这一层理解成：

**负责“怎么启动”。**

### 2. 原始数据处理流程

这一层回答的是：

**它怎么从历史会话里拿到可分析的内容。**

- 读取会话：`ingest.py`
- 项目归属判断：`attribution.py`
- 设计内容筛选：`filter.py`
- 主题聚类：`cluster.py`
- 规则提炼：`distill.py`
- 主流程编排：`pipeline.py`

这条线可以记成：

**会话 -> 找项目 -> 筛设计内容 -> 找重复主题 -> 提炼成规则洞察**

### 3. 智能体流程

这一层回答的是：

**为什么它不只是一个脚本，而是一个 Agent。**

- Agent 编排：`agent.py`
- 审校和裁剪：`review.py`
- LLM 润色：`llm.py`
- Skills 定义和选择：`skill_registry.py`、`skill_router.py`、`skill_executor.py`

这条线可以记成：

**先跑稳定流程 -> 选择合适 skill -> 质量审校 -> 策略重规划 -> 写入项目记忆 -> 可选润色 -> 生成草稿**

如果面试时要讲“智能体体现在哪”，核心就讲这几个文件。

### 4. 产出与落地流程

这一层回答的是：

**最后它会生成什么。**

- 报告输出：`report.py`
- 草稿生成：`draft_skill.py`
- 合并到已有技能目录：`apply_skill.py`
- 发布到暂存区：`publish_skill.py`

这条线可以记成：

**规则洞察 -> 报告 -> Skill 草稿 -> apply / publish**

### 5. Web 交互流程

这一层回答的是：

**设计师在前台点按钮之后，后面发生了什么。**

- 页面结构：`web/index.html`
- 页面逻辑：`web/app.js`
- 页面样式：`web/styles.css`
- Web 接口：`web.py`
- Web 操作层：`web_support.py`
- 后台任务状态：`run_jobs.py`

这条线可以记成：

**前台点启动 -> 调后端接口 -> 后台任务运行 -> 前台轮询状态 -> 展示草稿**

### 还有 3 个支撑文件

- 配置读取：`config.py`
- 数据结构定义：`models.py`
- 项目索引与汇总：`indexer.py`

如果你只想记一句最适合面试说的话，可以直接说：

**这个项目从文件结构上分成 5 层：启动入口、原始数据处理、智能体编排、产出落地、Web 交互。核心分析链路在 `pipeline.py`，智能体能力主要在 `agent.py + review.py + skill_router.py + tool_policy.py + memory.py + llm.py`，用户使用链路主要在 `web.py + web_support.py + web/app.js`。**

## 配置方式

复制示例配置：

```bash
cp .design-skill-miner.toml.example .design-skill-miner.toml
```

如果你用 Kimi / Moonshot：

```bash
export MOONSHOT_API_KEY="your-key"
```

本地配置中通常只需要写模型相关信息，不要把真实 key 写进去：

```toml
[agent]
review_min_score = 0.6
auto_prune = true
max_cycles = 3
memory_db_path = ".design-skill-miner/memory.db"
publish_requires_approval = true
```

`publish_requires_approval = true` 表示即使后端收到“自动发布”请求，也必须显式批准才会执行发布。

模型配置示例：

```toml
[llm]
enabled = false
provider = "openai-compatible"
base_url = "https://api.moonshot.ai/v1"
model = "moonshot-v1-8k"
api_key_env = "MOONSHOT_API_KEY"
```

## 开源前注意事项

- 不要提交 `.design-skill-miner.toml`
- 不要提交真实会话数据
- 不要提交真实 API Key
- 提交前先看一次 `git status`

更多细节见 [SECURITY.md](./SECURITY.md)。

## 开源发布清单（建议照着做一遍）

如果你准备把这个仓库公开，建议按下面顺序执行：

1. 先做安全检查：
- 确认 `.design-skill-miner.toml`、`.design-skill-miner/` 没有进入暂存区
- 运行密钥扫描：`rg -n "sk-[A-Za-z0-9]{10,}" .`
2. 再做基础可运行检查：
- `python3 -m compileall src/design_skill_miner`
- `node --check web/app.js`
3. 最后人工走一遍主流程：
- 选择项目 -> 启动智能体 -> 查看运行过程 -> 编辑草稿并保存 -> 发布到暂存区
4. 更新文档：
- README / ARCHITECTURE / HOW-TO / CHANGELOG 保持一致

更详细版本见 [docs/OPEN_SOURCE_CHECKLIST.md](./docs/OPEN_SOURCE_CHECKLIST.md)。
